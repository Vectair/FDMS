Pre-V1 Flite hardening order
1. Fix Admin backup/restore completeness

Add vectair_fdms_bookings_v1 to the session backup/restore key list and ensure bookings are included in data counts. Your verification showed bookings are stored under that key but omitted from SESSION_BACKUP_KEYS.

Acceptance test:

Create booking.
Create linked strip.
Export backup.
Clear/replace data.
Restore backup.
Confirm booking, linked strip, calendar/profile/reference data and reports data are all still present.
2. Replace all remaining native confirm() / alert() calls

Do this early because native dialogs are a packaged-app reliability risk. Your search confirmed remaining native dialog calls in booking, reports, reporting export fallback, and return-to-planned.

Replace with:

in-app confirmation modal for destructive actions
toast or inline validation message for simple success/failure
no browser-native dialogs anywhere in operational code

Specific areas:

Booking cancel linked strips.
Booking delete.
Calendar event delete.
Booking profile delete.
Reports hours save/clear validation.
Reports hours clear confirmation.
XLSX export fallback error.
Live Board return-to-planned confirmation.
3. Fix stale/prototype user-facing copy

Remove or rewrite any text that suggests the app is still a prototype. The worst confirmed example is the Booking page text saying the strip wiring is a future step, even though booking-to-strip creation now exists.

Search for terms like:

Select-String -Path .\src\**\*.* -Pattern "prototype|next step|actual Live Board|TODO|dev only|test|harness|temporary" -CaseSensitive:$false

Acceptance test:

No normal operator screen contains misleading development-era wording.
Browser-harness wording only appears when actually running in browser/dev mode.
4. Fix booking-to-strip link completeness

When a booking creates a movement strip, update the booking with the created strip ID immediately. At present, the movement gets bookingId, but the booking-side linkedStripId path is fragile/asymmetric.

This should be fixed before UI polish because the Booking workflow is one of the core operational flows.

Acceptance test:

Create booking and strip.
Open booking drawer/details.
Confirm linked strip is visible or traceable.
Cancel booking with linked strip.
Delete/cancel linked strip.
Run reconcile and confirm no stale links remain.

6. Resolve version/release hygiene

Your local check confirmed package.json, Cargo.toml, and tauri.conf.json are still at 0.9.0, while updater-test artefacts have used later versions.

Before V1:

Decide the next version path.
Align all version fields.
Ensure the GitHub latest release does not point operational users at a test build.
Keep test releases clearly marked or remove them from latest-channel exposure.
Confirm the app reports the expected version in Admin > Version & Updates.

Suggested scheme:

0.9.x = pre-V1 internal hardening builds
1.0.0-rc.1 = operational release candidate
1.0.0 = EGOW V1 deployment build

8. Clean Admin presentation for operational use

Admin can remain available, but it should not look like a developer console first and an operator tool second.

Work items:

12. Clarify Cancelled vs Deleted behaviour

The structure is good: Cancelled Sorties and Deleted Strips are separate, and Deleted Strips are recoverable for 24 hours.

Still, add a little more operator-facing explanation:

Cancelled = real sortie cancelled; retained as audit/history.
Deleted = accidental/wrong strip removal; recoverable for 24 hours.
Completed = normal flown/handled movement.

This reduces the chance that colleagues use Delete when they should Cancel.
